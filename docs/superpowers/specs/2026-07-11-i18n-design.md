# UI i18n (Chinese / English) — Design Spec

**Date:** 2026-07-11  
**Status:** Draft for review  
**Scope:** Bilingual UI only (first of three planned follow-ups: i18n → credential save → SFTP)

## Goals

- Let users switch the app UI between **Chinese (`zh`)** and **English (`en`)**.
- Default language on first launch: **Chinese**.
- Persist the user’s choice across restarts.
- Use a mature library: **i18next + react-i18next**.

## Non-Goals

- Additional locales beyond `zh` / `en` in this iteration
- Translating remote shell / SSH server output inside the terminal
- Translating OS-native dialog chrome that Electron/OS controls (where not practical)
- Credential persistence and SFTP (separate specs)

## Stack

| Piece | Choice |
|-------|--------|
| Library | `i18next`, `react-i18next` |
| Locale files | `src/renderer/src/i18n/locales/zh.json`, `en.json` |
| Init | `src/renderer/src/i18n/index.ts`, imported from `main.tsx` before render |
| Persistence | `userData/settings.json` via main-process store + IPC |

## Architecture

```
Renderer (React)
  useTranslation() / t('key')
  Language switcher → api.settings.get/set
        │
        │ IPC
        ▼
Main
  SettingsStore → %APPDATA%/.../settings.json
  { "language": "zh" | "en" }
```

- Locale JSON lives only in the renderer bundle (UI strings).
- Language preference is app settings owned by main (same pattern as hosts file: no secrets, simple JSON).

## Settings shape

```json
{
  "language": "zh"
}
```

- Valid values: `"zh"` | `"en"`.
- Missing / corrupt file → treat as `{ "language": "zh" }` without destroying a recoverable file when possible (same spirit as ConnectionStore: prefer safe defaults on missing file; on corrupt, surface error and do not blindly overwrite if we can avoid it — for v1 of settings, missing file creates default on first successful set).

## IPC

Extend shared types and preload `window.api`:

```ts
settings: {
  get: () => Promise<{ language: 'zh' | 'en' }>
  set: (patch: { language?: 'zh' | 'en' }) => Promise<{ language: 'zh' | 'en' }>
}
```

Channels (add to `IPC` const): `settings:get`, `settings:set`.

## UI behavior

1. App starts → load settings → `i18n.changeLanguage(language)`.
2. Language control in the host sidebar (header or footer): select **中文** / **English**.
3. On change → `api.settings.set({ language })` → `i18n.changeLanguage(language)` → UI re-renders via react-i18next.
4. No full app restart required.

## Strings in scope (first pass)

Cover all user-visible renderer copy currently hardcoded in English, including:

- Host list title, empty state, New / Connect / Edit / Delete, delete confirm
- Host form labels and buttons (name, host, port, username, auth method, browse, save, cancel)
- Main placeholder when no session
- Session tab close, disconnect/error banner, Reconnect
- Password modal title/labels/submit/cancel
- Host-key-changed confirm text
- Toast dismiss label and common error toasts originating from UI strings

Use nested keys, e.g. `hosts.title`, `hosts.empty`, `session.disconnected`, `auth.passwordTitle`.

## Error / edge cases

- Settings read failure → fall back to `zh`, show a short toast if useful.
- Settings write failure → keep previous language in memory, show toast.
- Unknown language value in file → fall back to `zh`.

## Testing

- Unit-test SettingsStore get/set defaults (main, pure file I/O) similar to ConnectionStore.
- Manual: switch zh↔en, restart app, confirm persistence; spot-check major screens.

## Follow-ups (out of this spec)

1. Credential save (password / key material) — security-focused design
2. SFTP — new feature module + UI
